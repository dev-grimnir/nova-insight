### Snapshot drill-down past-day end-time clamp fix
**File:** `src/views/nova-snapshot-chart.js`, `#mountTickClickTargets`

**Bug:** Clicking a day-tick on a snapshot drill-down hard-coded `drillEnd` to 23:59:59.999 of the clicked day. For *today*, that meant the analyzer credited the trailing gap from the last real event all the way to end-of-day in the last known status — inflating current-day uptime/downtime by however many hours remained.

**Fix:** Mirror the month-branch's clamp pattern on the day-branch. Append after the existing `drillStart`/`drillEnd` assignments in the `else` branch:

    if (drillStart.getTime() < startTime) drillStart = new Date(startTime);
    const upperMs = Math.min(endTime, Date.now());
    if (drillEnd.getTime() > upperMs) drillEnd = new Date(upperMs);

`Math.min(endTime, Date.now())` is defensive — `Date.now()` is universal upper bound, `endTime` keeps drill inside the snapshot. Past-day drills unaffected. Top-level snapshot was already fine (defaults `endDate` to `now`).

### Snapshot view refactor (3-commit set, single dev cycle)

Three structural changes shipped together. Visible duplication between the modal snapshot view and the report snapshot view was ~80% — same header structure, same drill wiring, same chart-lifecycle dance. The shared paradigm extracted into a new panel class; the two outer views shrank to chrome + composition.

#### Commit 1: Slice-not-fetch architecture
**File:** `src/controllers/nova-snapshot-controller.js`

Drill-downs no longer make HTTP calls. Initial load still does (via `buildData`), but `drillTo` now operates on the parent model's already-fetched events array.

New static `buildFromEvents(username, friendlyName, events, startDate, endDate)` — analyze-only path. Filters events to `[startDate, endDate]` (critical — see below), runs `NovaAnalyzer.computeMetrics`, returns a fresh `NovaSnapshotModel`.

`drillTo` updated:

    async drillTo(startDate, endDate) {
        const parent = this.historyStack[this.historyStack.length - 1];
        if (!parent) return null;
        const model = NovaSnapshotController.buildFromEvents(
            this.username, this.friendlyName, parent.events, startDate, endDate
        );
        if (!model) return null;
        this.historyStack.push(model);
        return model;
    }

Kept `async` keyword despite no awaits — preserves existing `await drillTo(...)` call sites unchanged.

`buildData` got the same defensive filter applied (the snapshot controller now consistently filters events to `[startDate, endDate]` before handing to analyzer, regardless of whether events came from HTTP or from a parent model).

### Snapshot drill-down 100% uptime fix (the bug that surfaced the slice-not-fetch architecture's need for window filtering)

After Commit 1 first deployed, every day-drilldown reported 100% uptime regardless of actual data. Root cause: `buildFromEvents` was handing the analyzer the entire parent event stream, but telling it `requestedStart`/`requestedEnd` = the narrow drill window. `#processAllEntries` accumulated `sessionSeconds` across the parent's full multi-day range; `#computeUptimeMetrics` divided by the narrow window's seconds; numerator > denominator; `Math.min(100, ratio)` clamped to 100.

Fix: filter `events` to `[startDate, endDate]` inside `buildFromEvents` before the analyzer call. The analyzer's `#computeLeadTime` injects its own boundary at `requestedStart`, so leading-state correctness doesn't require pre-window context.

### Dashboard 24-hour click 100% uptime fix
Same shape of bug, different source of extra events. The inline snapshot click via `open24HourSnapshot` reported 100% because `NovaHTTPController.paginateReportLogs`'s `#buildPaginationParams` silently overrides `shour`/`smin` to `00:00` when explicit args are null, widening the actual fetch from `[now-24h, now]` to `[start-of-day, now]`. The analyzer was then handed ~41 hours of events and told the window was 24.

Fix: apply the same defensive `[startDate, endDate]` filter to `buildData` (mirrors `buildFromEvents`). `buildData` and `buildFromEvents` now symmetric — both filter events to the requested window before the analyzer call, regardless of where events came from.

The HTTP controller's silent shour/smin override is the upstream cause and remains unfixed in this cycle (intentionally — flagged for memory carryover; see latent bugs below). The snapshot controller is now hardened against it.

## Architectural decisions confirmed this session

- **`NovaSnapshotChart.buildPeriods` stays internal.** Earlier in this session I argued for hoisting it to public static so the customer-view inline snapshot could share it. Grimnir scoped that out: `NovaInlineSnapshotView` works perfectly and is not being touched. The chart class continues to own period-building privately.
- **Boundary injection in `NovaAnalyzer.#computeLeadTime` is data, not presentation.** The injected opposite-status event at `requestedStart` is a valid anchor representing the modem's state entering the window. Parent models store events including the boundary (via `getEntries`); drill-downs inherit that, then the analyzer injects another boundary appropriate to the sub-window. Both layers' boundaries are correct for their respective windows.
- **No destroy methods anywhere in the snapshot system.** GC handles it.
- **The two snapshot view classes intentionally keep the same names (`NovaSnapshotView`, `NovaReportSnapshotView`) post-refactor.** The new shared class is `NovaSnapshotPanelView`.

## Latent bugs identified but not fixed this session

### `NovaHTTPController.#buildPaginationParams` silent time-of-day override
**File:** `src/controllers/nova-http-controller.js`

When `paginateReportLogs` is called with explicit `startDate`/`endDate` but null hour/min args, `#buildPaginationParams` silently forces `shour=0, smin=0` for the start and `ehour=23, emin=59` for non-today end dates. This overrides the time component of any explicit Date objects passed in. Callers that snap to midnight (the 3-day path) are unaffected; callers that don't (the 24-hour path) get a wider fetch window than requested.

**Fix when ready:** `#buildPaginationParams` should read `sDate.getHours()` / `sDate.getMinutes()` when explicit args are null, falling back to 0/0 only when no `startDate` was passed at all (the "fetch current month" entry path). Same for `eDate` and 23/59.

Snapshot controller is currently defended via window-filtering in `buildData`, but the bug remains for any other caller of `paginateReportLogs`. The `getLatestEntry` method in the same file calls it with `(username, startDate, now)` — verify whether the silent override matters for that caller before fixing.

### `getLatestEntry` backward-walk early-exit
**File:** `demo/nova-http-controller-synthetic.js`, `getLatestEntry`.

Loop walks events array backward. `t > now` does `continue`, `t < cutoff` does `break`. If a stream has `[past_event, future_event]` and the lookback window is narrow, the loop skips the future event, then breaks on the past event because it's older than the narrow cutoff — returning undefined even though widening the lookback would find it. Practically masked by progressive-lookback widening but semantically wrong. Should `continue` instead of `break`.

## Communication notes
- Hardware mode = design/planning, no code
- Software mode = code generation OK
- DON'T flip into software mode without being told. Wait for the explicit signal even after a long hardware-mode planning thread.
- User holds strong MVC opinions, calls out architectural violations sharply
- "Dev is where the work gets done; main is the stable baseline."
- Branch dev → fix one discrete problem → merge to main → delete dev → re-branch. Don't bundle unrelated fixes. (Exception: when multiple changes are tightly coupled and being tested as a bundle, ship together.)
- DO NOT pull `main.js` — production bootstrap is `scripts/nova-dashboard.user.js`
- **Don't use `web_fetch` on the repo. The fetch tool returns stale or wrong files repeatedly. This session it returned versions of `nova-snapshot-controller.js` and `nova-snapshot-view.js` that were both missing methods the live code calls, leading to bad advice. ALWAYS ask the user to paste the file. This is non-negotiable.**
- When asked for "the methods" or edits to a class, prefer separate code blocks per method, not one big dump.
- When asked to edit a class, **first show edits cleanly without iterating over the same block multiple times**. Each block in final form once.
- **NEVER deliver a code edit with `// ... rest unchanged` or any placeholder comment that elides surrounding code.** When showing a method body, give the complete method body. User pastes verbatim; placeholder comments will break syntax.
- When user is on dev/main/demo, REMEMBER which. Tailwind class availability differs:
  - Demo: precompiled tailwind.css with explicit safelist. Classes not in safelist produce no CSS.
  - Dev/main: runs as userscript on host page (admin.neonova.net). Tailwind from host CDN or none.
- When making CSS changes that depend on Tailwind utilities, prefer owning rules in inline `<style>` blocks rather than relying on utility classes that may or may not exist.
- **Don't trust the JS console in Tampermonkey for ground-truth diagnostics. `document.querySelectorAll(...)` and similar return values are unreliable from the user's console environment. Use the Elements inspector and DOM outer HTML pastes instead.**
- Top-level `await` at the browser console is unreliable — return values may show as `undefined` or fail silently. Don't trust console returns as ground truth; verify via in-code logging or DOM inspection.
- Long conversations degrade assistant accuracy — start fresh between phases.
- **"Blame the cache" is a lazy diagnosis.** Do not reach for it as first guess. Stale userscript bundles ARE a real failure mode, but check the live DOM (Elements panel) before invoking it.
- **Don't sneak previously-rejected suggestions back in.** When the user has said "no destroy methods" or "the inline view is out of scope," respect that decisively across every subsequent code block. Re-adding rejected patterns mid-refactor wastes the user's review time and erodes trust.
- **When a refactor is ready to commit, take the user at their word about the test plan.** If they say "I'll commit everything and test the bundle," don't keep proposing commit-by-commit sequencing.
- **ALWAYS deliver memory files inside a code block.** No exceptions.
