### Connecting... / Account Not Found bug fix (main → demo)
**File:** `src/controllers/neonova-dashboard-controller.js`, `updateCustomerStatus` method.

**Bug:** Three of 18 seeded demo customers (`pacific.st.node.04`, `bentleyville.tower.7`, `macgregor.relay` — all stable-business profiles) were being toasted as "not found in RADIUS" and removed. Manual add also failed. Triggered on demo but not on real RADIUS because real RADIUS reliably has *some* event in 11 months for any valid customer; the synthetic can legitimately produce empty streams for low-rate profiles.

**Root cause:** The `if (!latest)` branch lost its strict `latest === null` discriminator at some point. The synthetic correctly returns:
- `null` → username not in registry (genuinely not in RADIUS)
- `undefined` → registered but no events in window
- entry → found

But the controller treated both falsy values identically and stamped 'Account Not Found' on undefined. With low-rate stable-business RNG seeds producing zero past events in the 37-day stream window, every poll for those customers returned undefined for every lookback → stamped not found → toasted and removed.

**Fix:** Three-case branch:

    if (!latest) {
        if (latest === null) {
            // Registry miss — genuinely not in RADIUS
            customer.update('Account Not Found', 0);
        } else if (customer.lastEventTime !== null) {
            // Known existing customer with no recent events — preserve status, increment duration
            const eventDate = new Date(customer.lastEventTime);
            if (!isNaN(eventDate.getTime())) {
                const durationSeconds = Math.floor((Date.now() - eventDate.getTime()) / 1000);
                if (durationSeconds >= 0) customer.update(customer.status, durationSeconds);
            }
        }
        // else: never-seen customer, valid in registry, no events yet → leave as 'Connecting...'
        return;
    }

`null` vs `undefined` distinction now matters. Production is unaffected (real RADIUS rarely returns undefined for a valid customer) but the semantic is now correct.

### Synthetic stream floor (demo-only)
**File:** `demo/neonova-http-controller-synthetic.js`, `#generateStream`.

**Rule:** Every seeded demo customer MUST have at least one event with `dateObj.getTime() <= Date.now()` at stream-creation time. This is the dashboard's implicit invariant — without a past event, `getLatestEntry` returns undefined for every lookback and the customer is stuck on 'Connecting...' forever (after the controller fix above). The synthetic must satisfy what the dashboard expects of real RADIUS.

**Two failure modes the floor addresses:**
1. RNG produces zero disconnects across the whole 37-day window (low-rate stable-business)
2. RNG produces events that all land in the future portion (after `demoEpoch`)

**Fix:** Block inserted in `#generateStream` after the switch, before `events.sort(...)`:

    const nowMs = Date.now();
    const hasPastEvent = events.some(e => e.dateObj.getTime() <= nowMs);
    if (!hasPastEvent) {
        const anchorAgeMs = (7 + rng() * 7) * 24 * 3600 * 1000;
        const anchorMs = epoch.getTime() - anchorAgeMs;
        events.push({ dateObj: new Date(anchorMs), status: 'Start' });
    }

Same rng instance preserves determinism per username. Start event so customer reads as Connected on initial poll (appropriate for stable-business). Floor runs for all cases uniformly; override profiles already have past events but the floor is cheap insurance.

**Carryover rule:** Any new profile type added later must produce at least one past event. Demo profiles fit the dashboard's invariants, not the other way around.

### `getLatestEntry` backward-walk early-exit
**File:** `demo/neonova-http-controller-synthetic.js`, `getLatestEntry`.

Loop walks events array backward. `t > now` does `continue`, `t < cutoff` does `break`. If a stream has `[past_event, future_event]` and the lookback window is narrow, the loop skips the future event, then breaks on the past event because it's older than the narrow cutoff — returning undefined even though widening the lookback would find it.

In practice the progressive lookback in `updateCustomerStatus` eventually widens enough that the past event falls inside `[cutoff, now]` and gets returned. But the early-break is semantically wrong; should `continue` instead of `break`, paying the full scan. Streams are small enough that the cost is trivial.
